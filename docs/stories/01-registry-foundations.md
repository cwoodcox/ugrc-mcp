# Story 1 — Registry & data pipeline foundations

**One-liner:** Stand up the v0.2 registry shape (`ORGS`, types, generated `sgid.ts`) and the offline sync pipeline that builds it from `agrc/sgid-index` + per-layer `?f=pjson` + a hand-curated enrichment sidecar.

## Goal

This story is the load-bearing first slab of v0.2. Stories 2 (query primitives), 3 (discovery factory + `find_layer`), and 5 (polish/cutover) all import from `src/registry/*`. Until those types exist and the generated `sgid.ts` is on disk, nothing else can typecheck or return a real catalog. The story must land first and must land complete: types frozen, `ORGS` defined, sync script idempotent, generated registry committed, v0.1's seven layers (and their hand-learned notes) absorbed into the new enrichment sidecar so we don't lose institutional memory in the migration. No tool registration, no wire-up — just the data layer.

## What this story IS

- Hand-write `src/registry/orgs.ts` with the single v0.2 entry (`ugrc → "99lidPhWCzftIe9K"`) and a forward-compatible schema.
- Define the v0.2 registry types (`Org`, `SgidLayer`, `SgidCategory`, the shape of the generated `SGID` export) and export them from `src/registry/types.ts`.
- Write `scripts/sync-sgid-registry.ts` — the offline fetch + transform + write pipeline.
- Write `scripts/enrichment.ts` — the hand-maintained sidecar of `useful_fields` / `gaps` / `caveats` / `time_field` overrides, seeded with v0.1's seven entries plus the cadastre/water category-level gap notes from plan.md.
- Run the sync once; commit the generated `src/registry/sgid.ts` (~235 layers across 20 categories).
- Add an `npm run sync-sgid-registry` script to `package.json`.

## What this story IS NOT

- **Not registering any tools.** `src/mcp.ts` is untouched; no `list_<category>` tool factory; no `find_layer`; no new `arcgis_query`. Those land in S2 and S3.
- **Not removing or editing `src/registry.ts`.** v0.1 keeps working during the migration; S5 deletes the old file at cutover.
- **Not editing `src/arcgis.ts`.** No rename to `arcgis-client.ts`, no `{ org, layer }` resolver in the client. S2 does the promotion.
- **Not building `find_layer` or any Hub Search code.** That's Tier 2 / S3.
- **Not implementing `describe_layer`, `list_capabilities`, or `McpServer.instructions`.** S5.
- **Not building `mapserv.ts` or wiring `UGRC_API_KEY`.** S4.
- **Not scheduling sync via GitHub Actions.** Manual cadence per plan.md §"Sync cadence".
- **Not fetching the registry at runtime.** Reproducibility / cold-start / outbound-budget reasons in plan.md §"Why not fetch at runtime".

## Dependencies

- **Depends on:** nothing. This is the foundation.
- **Unblocks:**
  - **S2** — imports `ORGS` + types to resolve `{ org, layer }` → URL in the new `arcgis_query` / `arcgis_aggregate`.
  - **S3** — imports the generated `SGID` data to emit one `list_<category>` tool per category and to power `describe_layer`'s cataloged-lookup branch.
  - **S5** — uses `ORGS` + `SGID` to populate `list_capabilities` and the description nudges.
- **Parallel with:** **S4** (mapserv) — no shared files.

### 1. Types — `src/registry/types.ts`

Export at least:

- `type OrgHandle = "ugrc"` (extend as needed in v0.3).
- `interface Org { name: string; agol_id: string; url_base: string; }`
- `interface FieldSummary { name: string; type: string; alias?: string; }`
- `interface SgidLayer { org: OrgHandle; layer: string; name: string; steward?: string; service_path: string; geometry_type?: string; last_edit_date?: string | null; max_record_count?: number; extent?: { xmin: number; ymin: number; xmax: number; ymax: number; spatialReference?: { wkid?: number; latestWkid?: number } }; field_summary?: FieldSummary[]; useful_fields?: readonly string[]; gaps?: readonly string[]; caveats?: readonly string[]; time_field?: string; }`

- `interface EnrichmentEntry { useful_fields?: readonly string[]; gaps?: readonly string[]; caveats?: readonly string[]; time_field?: string; manual_category?: string; }` — keyed in `enrichment.ts` by `${org}/${layer}`.

### 4. Sync script — `scripts/sync-sgid-registry.ts`

Implements the flow in plan.md §"Sync flow":

1. **Fetch upstream.** `GET` `https://raw.githubusercontent.com/agrc/sgid-index/main/src/data/downloadMetadata.ts`. **Pin a commit SHA**, not `main`, to satisfy plan.md risk #1 (`downloadMetadata.ts` upstream stability). Store the SHA as a top-of-file constant; bumping it is a deliberate edit.
2. **Parse.** The file is data-only TypeScript. Use regex / `ts.transpileModule` / a tiny `eval`-in-VM — whichever is robust to the file shape today. Output: array of `{ itemId, featureServiceId?, layerId?, openSgid?, name, steward? }`.
3. **Filter.** Drop entries without `(openSgid prefix && featureServiceId)`. Reroute entries that match a `manual_category` in `ENRICHMENT` keyed by AGOL `itemId` or by derived `org/layer` key. Drop entries whose `featureServiceId` doesn't resolve to a known `ORGS` entry (v0.2: only `99lidPhWCzftIe9K`). Log every drop.
4. **Compute the layer key.** Stable, short, snake_case, unique within the org. Suggested rule: slugify upstream `name`, with hand-overrides in enrichment to align with v0.1 keys (`wrlu`, `parcels_lir`, `parcels_basic`, `watersheds_huc12`, `ag_protection_areas`, `land_ownership`, `solar_zones`). Document the rule at the top of the script.
5. **Per-layer pjson fetch.** For each surviving entry, `GET <service_url>?f=pjson` (parallel, batched ~10 at a time, 5xx retry with backoff — reuse the pattern from `src/arcgis.ts` `arcgisJson`). Extract `editingInfo.dataLastEditDate` (format to ISO date), `geometryType`, `maxRecordCount`, `extent`, and a `field_summary` of just `{ name, type, alias }` per field. Pjson failure is **non-fatal**: log it, set `last_edit_date: null`, omit `field_summary`, keep the layer.
6. **Merge enrichment.** For each layer, look up `ENRICHMENT["ugrc/<key>"]` and copy `useful_fields`, `gaps`, `caveats`, `time_field` onto the `SgidLayer`.
7. **Group by category** (`openSgid` prefix, or `manual_category` from enrichment). Apply `CATEGORY_OVERRIDES` blurb + `category_gaps`.
8. **Write.** Render `src/registry/sgid.ts` as a single `export const SGID: SgidRegistry = { ... } as const satisfies SgidRegistry;`. Sort categories alphabetically and layers within each category by key for stable diffs. Prepend a generated-file banner: `// AUTOGENERATED by scripts/sync-sgid-registry.ts from agrc/sgid-index@<SHA>. Do not edit by hand.`
9. **Print a summary.** Layer count per category, total, count of pjson failures, count of dropped upstream entries, count of layers placed via `manual_category`.
10. **Exit 0.** Maintainer reviews `git diff` and commits.

### 5. Wire `npm run sync-sgid-registry`

Add to `package.json` `scripts`. Use a runner (`tsx`, `ts-node`) that doesn't require Wrangler — this is a plain Node script.

### 6. Run it; commit the output

`npm install && npm run sync-sgid-registry`, then `git add src/registry/ scripts/ package.json package-lock.json` and commit on the story branch.

## Risks & mitigations

| Risk | Mitigation in this story |
| --- | --- |
| **R1 — `downloadMetadata.ts` upstream rename/move.** | Sync script pins a known-good commit SHA constant. Bumping the SHA is a deliberate edit. Failure is loud (fetch 404). |
| **R5 — SGID URL drift (silent renames).** | Sync script logs every dropped/skipped upstream entry and every pjson failure. Layer keys are stable across syncs; the diff in `src/registry/sgid.ts` will surface a URL change at PR review. |
| **Parsing `downloadMetadata.ts` is fragile.** | File is data-only TS; pin a SHA *and* validate parse output with a top-of-script assertion on minimum entry count (~360) — fail fast if upstream restructures. |
| **Per-layer pjson fetch is slow.** | Batched parallel fetches (~10 at a time) with 5xx retry mirroring `src/arcgis.ts`. Failures are non-fatal — layer keeps upstream metadata only. |
| **Layer keys collide with v0.1.** | Hand-override keys in `enrichment.ts` so the v0.1 seven (`wrlu`, `parcels_lir`, …) survive verbatim. Acceptance criterion #5 enforces this. |
| **Drift between this story's types and S2/S3 needs.** | Types file is small and final in this story; S2/S3 import it as-is. If a downstream story needs a type tweak, it's a single shared edit, not a cross-cutting rewrite. |
