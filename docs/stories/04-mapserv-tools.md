# Story 4 — mapserv tools + API key wiring

Implement 7 wrappers over the UGRC Web API (https://api.mapserv.utah.gov/) plus the `UGRC_API_KEY` Worker secret that gates them. Story 5 registers them; this story ships the code + the binding.

---

## Goal

Geocoding and mapserv-side SGID search are independently valuable (forward/reverse address, milepost ↔ point, SQL-like attribute search, table/field enumeration), and — more importantly for v0.2's discovery thesis — they give the agent seven `verb_noun`-named tools that pattern-match obvious user intents. *"What's the address at X, Y?"* should fire `reverse_geocode` on first read of the tool list, not trigger a curl-writing exercise. This story produces the wrappers + secret wiring; Story 5 hangs them off `mcp.ts`.

---

## What this story IS

- 7 exported async wrapper functions in `src/tools/mapserv.ts`, one per mapserv endpoint in plan.md §"mapserv (7 tools)".
- A `UGRC_API_KEY` Worker secret declared in `wrangler.jsonc` so `npm run cf-typegen` emits it on `Env`.
- A single registration function exported from `src/tools/mapserv.ts` that Story 5 calls from `mcp.ts` to attach all 7 tools to the `McpServer` instance.
- The "missing secret" precheck per tool that returns the documented error string verbatim.
- README note documenting `wrangler secret put UGRC_API_KEY`.

## What this story IS NOT

- **NOT** mcp.ts registration. Story 5 imports the registration function this story exports; no edit to `mcp.ts` happens here.
- **NOT** registry work. No `ORGS`, no `SGID_CATEGORIES`, no sync script. Story 1 owns those.
- **NOT** `arcgis_query` / `arcgis_aggregate` / `arcgis_raw`. Story 2 owns those.
- **NOT** rate-limit enforcement. Risk #2 (no documented mapserv rate limit) is deferred unless abused — README nudges self-deploy / Cloudflare Access.
- **NOT** a replacement for `arcgis_query`. `search_sgid_via_mapserv` overlaps `arcgis_query` deliberately — it's kept for query shapes AGOL `/query` can't express 1:1 (per plan.md). The descriptions must not pitch it as the default read path.
- **NOT** a caching layer. Mapserv responses are read-through.

## Dependencies

- **Upstream:** none. S4 has no code dependency on S1/S2/S3 and can begin on day 1 in parallel with them (per epic §"Critical path").
- **Downstream:** unblocks S5, which registers these tools and ships discovery polish.

## Tool inventory

All 7 wrappers default `spatialReference=4326` in our wrapper layer (mapserv's native default is UTM 12N — wrong for an LLM). All 7 short-circuit with the documented "set UGRC_API_KEY" error string if the secret is unset.

| Tool | Mapserv endpoint | Inputs | Output shape | Purpose |
| --- | --- | --- | --- | --- |
| `geocode_address` | `GET /api/v1/geocode/{street}/{zone}` | `street`, `zone` (city OR zip), optional `spatial_reference` (default `4326`), optional `acceptScore` | `{ matchAddress, score, locator, location: { x, y }, addressGrid }` | Address → coordinates. |
| `reverse_geocode` | `GET /api/v1/geocode/reverse/{x}/{y}` | `x`, `y`, optional `spatial_reference` (default `4326`), optional `distance` (default 5 m) | `{ address, grid, score, location, distance }` | Coordinates → nearest street address. |
| `geocode_milepost` | `GET /api/v1/geocode/milepost/{route}/{milepost}` | `route` (UDOT route id), `milepost`, optional `spatial_reference` (default `4326`), optional `side` | `{ route, milepost, location, increment }` | UDOT route + milepost → coordinates. |
| `reverse_milepost` | `GET /api/v1/geocode/reversemilepost/{x}/{y}` | `x`, `y`, optional `spatial_reference` (default `4326`), optional `route` (filter to one route) | `{ route, milepost, location, offsetMeters }` | Coordinates → nearest UDOT route + milepost. |
| `search_sgid_via_mapserv` | `GET /api/v1/search/{tableName}/{returnValues}` | `table_name`, `return_values` (comma-separated columns), optional `predicate` (SQL-ish WHERE), optional `geometry` (WGS84 GeoJSON), optional `spatial_reference` (default `4326`), optional `attributeStyle` | `{ results: [...rows] }` | Mapserv's SQL-like SGID search. Kept for query shapes AGOL `/query` can't express 1:1 (overlaps `arcgis_query` by design). |
| `list_sgid_tables` | `GET /api/v1/info/featureClassNames/{category?}` | optional `category` (single SGID category like `cadastre`) | `{ result: string[] }` | Enumerate mapserv-known SGID table names. |
| `list_sgid_fields` | `GET /api/v1/info/fieldnames/{tableName}` | `table_name` | `{ result: string[] }` | Enumerate columns of a single SGID table. |

For every entry above, the wrapper:

1. Appends `apiKey=<env.UGRC_API_KEY>` as a query param (not a header — mapserv accepts both, query is what its docs use).
2. Defaults the `spatialReference` query param to `4326` unless the caller passes another EPSG code.
3. Path-encodes user inputs (`encodeURIComponent`) before substitution into the URL template.
4. Returns the parsed JSON `body.result` (or `body` for endpoints with no `result` wrapper) — preserving mapserv's shape so the model can read it directly.

## Task list

1. **Declare the secret in `wrangler.jsonc`.** Cloudflare secrets aren't put under `vars` (that commits them). The accepted pattern is: leave the production value to `wrangler secret put UGRC_API_KEY`, and add a `.dev.vars` entry for local `wrangler dev`. Add `.dev.vars` to `.gitignore` if not already there. To get `UGRC_API_KEY: string` onto the `Env` type via `npm run cf-typegen`, the cleanest approach is to set the secret on the deployed worker once (typegen reads from the deployed worker config). Document both paths in the README subsection.
2. **Regenerate types.** `npm run cf-typegen` to refresh `worker-configuration.d.ts`. Verify `UGRC_API_KEY: string` shows up on `Cloudflare.Env`.
3. **Create `src/tools/mapserv.ts`.** Module-level constant `MAPSERV_BASE = "https://api.mapserv.utah.gov"`. Module-level constant `MISSING_KEY_MSG = "Set UGRC_API_KEY via \`wrangler secret put UGRC_API_KEY\`. Request a key at developer.mapserv.utah.gov."` — used verbatim by every tool's precheck so the string stays in one place.
4. **Implement a shared HTTP helper** in the same file — `mapservGet<T>(path: string, query: Record<string, string | number | undefined>, env: Env): Promise<T>`. Mirrors the conventions in `src/arcgis-client.ts` (post-rename per S2): native `fetch`, `AbortSignal.timeout(30_000)`, 5xx retry with 100/200 ms backoff (same shape as `arcgis.ts:arcgisJson`). Url-encodes `query`. Throws on non-2xx with the mapserv error body included; passes 4xx straight through (mapserv returns `{ message: "..." }`).
5. **Implement each of the 7 wrappers** as small async functions that:
   - Take a typed input object (see "Function signatures" below).
   - Take `env: Env` as the last argument.
   - First check `env.UGRC_API_KEY`; if falsy, return/throw the documented error verbatim. The exact decision (return vs throw) is set by how Story 5 wires this into the MCP tool handler — recommend `throw new Error(MISSING_KEY_MSG)` so MCP returns it as a tool error and the client surfaces it.
   - Build the URL by template + `encodeURIComponent` on every interpolated segment.
   - Default `spatialReference: 4326` for the four endpoints that take it.
   - Call `mapservGet`, return the parsed body.
6. **Implement `registerMapservTools(server: McpServer, env: Env): void`** that calls `server.registerTool` (or the SDK's equivalent) seven times with: `name`, `description`, `inputSchema` (zod), and a handler closing over `env`. Story 5 imports and calls this from `mcp.ts`. Keeping registration in this file means S4 can ship without touching `mcp.ts`; S5's diff is a single import + call.
7. **Secret hygiene.** Never echo `UGRC_API_KEY` (or any value derived from it) in tool responses, error messages, or thrown errors. The HTTP helper logs only the URL *without* `apiKey`. Confirm with grep before commit.
8. **README update.** Add a "Configuration" subsection:
   ```
   ## Configuration

   Set the mapserv API key once per environment:

   - Local dev: add `UGRC_API_KEY=...` to `.dev.vars` (gitignored).
   - Deployed: `wrangler secret put UGRC_API_KEY`

   Request a key at https://developer.mapserv.utah.gov. Tools that need it
   short-circuit with a clear message if the secret is unset.
   ```

## Tool description copy

The `verb_noun` names carry most of the discovery weight. Descriptions stay tight and mention the WGS84 default; they do not duplicate parameter docs (zod schemas + MCP Inspector handle that).

- **`geocode_address`** — *"Forward geocode a Utah street address to a coordinate. Returns location, match score, and address grid. Coordinates returned in WGS84 (EPSG:4326) by default."*
- **`reverse_geocode`** — *"Reverse geocode a WGS84 coordinate to the nearest Utah street address. Returns the matched address, grid, score, and offset distance."*
- **`geocode_milepost`** — *"Resolve a UDOT route + milepost to a WGS84 coordinate. Use for route-anchored locations (incident reports, asset inventories)."*
- **`reverse_milepost`** — *"Find the nearest UDOT route + milepost to a WGS84 coordinate. Optionally constrain to a specific route."*
- **`search_sgid_via_mapserv`** — *"SQL-style attribute search against a single SGID table via the mapserv `/search` endpoint. Use for predicates AGOL `/query` can't express 1:1; for typical reads, prefer `arcgis_query`."*
- **`list_sgid_tables`** — *"List mapserv-recognized SGID table names, optionally filtered to one SGID category. Pair with `list_sgid_fields` then `search_sgid_via_mapserv`."*
- **`list_sgid_fields`** — *"List the columns of a single SGID table as known to mapserv. Use to build `return_values` and `predicate` for `search_sgid_via_mapserv`."*

## Function signatures

Sketches — zod schemas live in the same file so they're shared with the tool registration:

```ts
import { z } from "zod";
import type { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";

const SrSchema = z.number().int().positive().default(4326);

export const GeocodeAddressInput = z.object({
  street: z.string().min(1),
  zone: z.string().min(1), // city or 5-digit zip
  spatial_reference: SrSchema,
  accept_score: z.number().int().min(0).max(100).optional(),
});
export type GeocodeAddressInput = z.infer<typeof GeocodeAddressInput>;
export async function geocodeAddress(
  input: GeocodeAddressInput,
  env: Env,
): Promise<unknown>;

export const ReverseGeocodeInput = z.object({
  x: z.number(),
  y: z.number(),
  spatial_reference: SrSchema,
  distance: z.number().positive().default(5), // meters
});
export async function reverseGeocode(
  input: z.infer<typeof ReverseGeocodeInput>,
  env: Env,
): Promise<unknown>;

export const GeocodeMilepostInput = z.object({
  route: z.string().min(1), // UDOT route id, e.g. "0015"
  milepost: z.number().nonnegative(),
  spatial_reference: SrSchema,
  side: z.enum(["increasing", "decreasing"]).optional(),
});
export async function geocodeMilepost(
  input: z.infer<typeof GeocodeMilepostInput>,
  env: Env,
): Promise<unknown>;

export const ReverseMilepostInput = z.object({
  x: z.number(),
  y: z.number(),
  spatial_reference: SrSchema,
  route: z.string().min(1).optional(), // optional route filter
});
export async function reverseMilepost(
  input: z.infer<typeof ReverseMilepostInput>,
  env: Env,
): Promise<unknown>;

export const SearchSgidViaMapservInput = z.object({
  table_name: z.string().min(1),
  return_values: z.array(z.string().min(1)).min(1), // comma-joined into URL
  predicate: z.string().optional(),
  geometry: z.unknown().optional(), // GeoJSON WGS84
  spatial_reference: SrSchema,
  attribute_style: z.enum(["identical", "lower", "upper"]).optional(),
});
export async function searchSgidViaMapserv(
  input: z.infer<typeof SearchSgidViaMapservInput>,
  env: Env,
): Promise<unknown>;

export const ListSgidTablesInput = z.object({
  category: z.string().optional(),
});
export async function listSgidTables(
  input: z.infer<typeof ListSgidTablesInput>,
  env: Env,
): Promise<unknown>;

export const ListSgidFieldsInput = z.object({
  table_name: z.string().min(1),
});
export async function listSgidFields(
  input: z.infer<typeof ListSgidFieldsInput>,
  env: Env,
): Promise<unknown>;

export function registerMapservTools(server: McpServer, env: Env): void;
```

Story-specific (derived from plan.md):

- **F8.1** — The missing-key error string matches plan.md §"mapserv (7 tools)" exactly: *"Set UGRC_API_KEY via `wrangler secret put UGRC_API_KEY`. Request a key at developer.mapserv.utah.gov."*
- **F8.2** — `search_sgid_via_mapserv` ships even though `arcgis_query` overlaps. Its description steers default reads to `arcgis_query`.
- **F8.3** — Each wrapper path-encodes inputs before URL interpolation (mapserv paths take raw text like street names).
- **F8.4** — `registerMapservTools` is the sole side-effecting export from the module; pure function exports stay testable.

## Non-functional requirements

From epic §"Non-functional requirements":

- **N8** — `UGRC_API_KEY` is a Worker secret (not committed). `.dev.vars` is gitignored. README documents `wrangler secret put`.
- **N9** — All HTTP follows the v0.1 invariants: native `fetch`, `AbortSignal.timeout(30_000)`, 5xx retry with backoff. Reuse the shape from `arcgis.ts` (S2 renames it to `arcgis-client.ts` but the conventions don't change); a tiny per-file `mapservGet` is cleaner than coupling to the ArcGIS client since mapserv has different error semantics (no HTTP-200-with-error pattern; it uses proper 4xx).
- **N10** — `npm run typecheck` passes at end of story. No half-typed handlers.

Story-specific:

- API key never appears in returned objects, thrown error messages, console logs, or stringified Request URLs sent to `console.*`. Grep for `apiKey` and `UGRC_API_KEY` in the diff before committing.
- Request timeout is 30 s exactly (matching v0.1).
- Retries on 5xx only; 4xx (including 401/403 from a bad key) are surfaced immediately so the user knows to rotate the key.

## Acceptance criteria

1. `npm run typecheck` passes after `npm run cf-typegen`.
2. `npm run cf-typegen` produces `UGRC_API_KEY: string` on `Cloudflare.Env` in `worker-configuration.d.ts`.
3. Calling any of the 7 wrappers with `env.UGRC_API_KEY` unset returns the exact string *"Set UGRC_API_KEY via `wrangler secret put UGRC_API_KEY`. Request a key at developer.mapserv.utah.gov."*
4. With a valid key set, `reverseGeocode({ x: -111.8910, y: 40.7608 }, env)` returns a result containing a Salt Lake City address (street + city = `SALT LAKE CITY`).
5. With a valid key set, `geocodeAddress({ street: "326 East South Temple", zone: "Salt Lake City" }, env)` returns a `location` whose `{x, y}` is within ~50 m of (-111.879, 40.770).
6. Epic AC **A2** — *"What's the street address at 40.7608° N, 111.8910° W?"* — is answerable end-to-end via `reverse_geocode` once Story 5 registers the tools and a key is configured.
7. `registerMapservTools(server, env)` registers exactly 7 tools on `server`; the names match the table above verbatim.
8. No occurrence of the API key value appears in any function's return payload or thrown error message (manual grep + smoke test).

## Risks & mitigations

- **R2 (epic) — No documented rate limit on mapserv.** Hosted-instance abuse could get our key throttled or revoked. README nudges heavy users to self-deploy with their own key; if abuse becomes real, add a Cloudflare WAF rule or move the endpoint behind Cloudflare Access. No code-level rate limit in v0.2.
- **Secret leakage.** `apiKey` is appended to query strings on every outbound request. Never log the URL with that param attached; never echo `env.UGRC_API_KEY` in error messages, return values, or `console.*`. The `mapservGet` helper strips `apiKey` before any debug logging.
- **Mapserv schema drift.** Mapserv occasionally renames response fields. The wrappers pass body through largely unmodified; if a field rename breaks a downstream consumer, the fix is local to one wrapper.
- **`search_sgid_via_mapserv` becoming the default read path.** Its description has to steer reads back to `arcgis_query`. Watch for this in acceptance-test transcripts once Story 5 lands.
